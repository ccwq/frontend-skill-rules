# frontend-tech-stack

## 定位与触发场景

这是前端工程的**技术选型与依赖边界铁律**。

高优先级目标：

- 技术栈跟随当前项目动态推导。
- 不为了规范强制迁移语言。
- 节省 token，人类阅读友好。
- 只管技术选型，不管目录命名、编码风格或视觉品牌。

在以下场景必须使用：

- 开始任何前端开发、重构、生成页面或生成组件之前。
- 准备新增 npm/pnpm 依赖之前。
- 准备引入新的 UI 库、状态管理、路由方案、样式体系、图表库之前。
- 从原型或规范推导当前项目技术栈时。
- 用户要求“技术栈规范 / 技术选型 / 依赖边界 / 前端栈约束”。

## 三层必跑协议

当前端开发/修改/审查任务涉及 Vue 页面、组件、API、Mock、样式或交互时，默认依次执行：

1. `frontend-tech-stack`
2. `frontend-project-structure`
3. `frontend-code-style`

若用户只显式调用当前 skill，则只执行当前层，但最终报告必须提醒：**三层必跑协议未完整执行**。

三层规则冲突时，优先级固定为：

1. `frontend-tech-stack`：技术选型、依赖边界、构建体系优先。
2. `frontend-project-structure`：目录结构、目录命名、文件命名次之。
3. `frontend-code-style`：具体编码风格、错误处理、资源清理、交互完整性再次。

任一自检项 `FAIL`，默认阻塞交付；必须继续修复到 `PASS`，或记录“用户确认例外”的原因。

## 强制执行流程

1. 开发前先确认当前项目技术栈来源。
2. 优先读取当前项目事实：`package.json`、锁文件、Vite 配置、UnoCSS 配置、路由/Pinia/Element Plus 使用痕迹、项目规范文档、原型目录。
3. 将“规范要求”和“现有工程事实”合并成当前任务的技术栈边界。
4. 若规范与现有工程冲突，先报告冲突，不擅自替换技术栈。
5. 未经用户确认，不新增与既有方案重叠的新依赖。
6. 完成后逐项自检；有 `FAIL` 不得声称完成。

## 动态推导原则

`frontend-tech-stack` 不是永久写死的全局技术栈。它必须跟随不同项目动态变化。

推导顺序：

1. 用户明确要求优先。
2. 当前项目规范文件优先。
3. 当前项目 `package.json`、锁文件和配置文件次之。
4. 当前项目原型、静态 demo、设计稿体现的能力需求再次。
5. 通用最佳实践最后。

如果换到另一个项目，必须重新根据该项目的规范、原型和依赖推导技术栈，不得机械套用本项目基线。

## 当前项目技术栈基线

当前项目的前端技术选型由项目规范与 `final-prototype` 能力需求共同推导。

### 核心框架

- UI 框架：Vue 3。
- 开发语言：跟随当前项目现状，可使用 JavaScript 或 TypeScript；本 skill 不强制 TypeScript，也不要求为了规范迁移已有 JavaScript 项目。
- 构建工具：Vite。
- 包管理：pnpm。

禁止未经确认改用 React、Angular、Svelte、Nuxt、Webpack、npm/yarn 作为主方案。

### 路由

- 路由库：vue-router。

禁止手写集中式路由数组作为主要路由方案。

### 状态管理

- 默认状态管理：pinia。

禁止未经确认引入 Vuex、Redux、Zustand、MobX 等替代 pinia 的状态管理方案。

### UI 组件库

- 默认 UI 库：Element Plus。
- 图标组件按需使用：`@element-plus/icons-vue`。

禁止同时引入 Ant Design Vue、Naive UI、Vuetify、Arco Design 等与 Element Plus 重叠的主 UI 组件库，除非用户明确要求迁移或兼容。

### 图标方案

- 默认图标集：`@iconify-json/mdi`。
- 默认使用方式：UnoCSS presetIcons，形如 `<i class="i-mdi-xxx" />`。

`final-prototype` 中的 Material Symbols 只作为原型表达，不作为 Vue 工程默认依赖。除非用户明确要求，不新增 Material Symbols 字体依赖。

### 样式方案

- 原子化样式：UnoCSS。
- CSS 预处理器：Less。
- 组件局部样式：Vue SFC style + Less 或项目现有写法。

禁止未经确认引入 Tailwind CSS、Sass/SCSS、styled-components、CSS-in-JS 等与既有样式体系重叠的方案。

### 图表与可视化

根据原型中的数据看板、统计卡片、趋势、环图、瀑布流等能力，允许按需使用：

- echarts。
- echarts-gl。
- echarts-liquidfill。

原则：

- 普通统计卡片和简单进度条不必强行引入图表库。
- 复杂趋势、图形化分析、3D 或液态图按需引入。
- 引入前确认项目是否已有图表封装。

### 网络请求

当前技术栈规范只确定“请求必须统一封装”，不在本 skill 中指定具体 API 函数语义。

如项目已有 axios/request 封装，必须复用；没有时需先确认是否引入 axios 或使用既有 fetch 封装。

### 原型能力对技术栈的影响

`final-prototype` 体现了以下能力需求，可作为技术选型参考：

- 企业后台布局。
- 数据看板。
- 通知和消息中心。
- 弹窗、表格、筛选、状态标签。
- 多主题切换。
- AI 助手入口。
- 高信息密度管理界面。

这些能力影响组件库、图标、图表、状态管理、样式体系选型，但不代表要照搬原型中的原生 JS、原生 CSS 或 Material Symbols 依赖。

## 新增依赖规则

新增依赖前必须回答：

1. 当前技术栈是否已有等价能力？
2. Element Plus、VueUse、UnoCSS、ECharts 是否已能满足？
3. 新依赖是否与现有方案重叠？
4. 新依赖是否会增加维护成本或风格割裂？
5. 用户是否明确批准？

未经确认，禁止新增：

- 第二套 UI 库。
- 第二套路由方案。
- 第二套状态管理。
- 第二套原子化 CSS 框架。
- 与现有图标方案冲突的图标字体。
- 大型可视化库替代 ECharts。

## 禁止事项

- 禁止擅自改用其他主框架。
- 禁止绕开 Vite 构建体系引入不兼容配置。
- 禁止同时维护多套 UI 库。
- 禁止同时维护多套状态管理。
- 禁止同时维护多套样式体系。
- 禁止把原型依赖误判为工程依赖。
- 禁止为单个页面需求新增重型依赖而不评估替代方案。
- 禁止未检查当前项目事实就机械套用其他项目技术栈。
- 禁止为了规范强制迁移 JavaScript / TypeScript 语言选择。

## 自检清单

执行结束必须逐项检查：

- [ ] 已确认当前项目技术栈来源：规范、依赖、配置、原型或用户要求。
- [ ] 开发语言跟随项目现状，未强制迁移。
- [ ] 主框架为当前项目确认的框架，未擅自替换。
- [ ] 路由方案符合当前项目约定。
- [ ] 状态管理方案符合当前项目约定。
- [ ] UI 组件库符合当前项目约定。
- [ ] 图标方案符合当前项目约定。
- [ ] 样式方案符合当前项目约定。
- [ ] 图表/可视化库按需使用，没有过度引入。
- [ ] 未新增与既有方案重叠的依赖。
- [ ] 如需新增依赖，已说明原因并等待/获得用户确认。
- [ ] 三层必跑协议状态已说明。
