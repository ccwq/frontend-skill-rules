# frontend-reuse-governance

## 定位与触发场景

这是前端工程的**复用治理规范**。

高优先级目标：

- 开发前先按层级查找可复用资产，避免重复创建功能相近的组件或逻辑。
- 能复用先复用，能小范围扩展再扩展；超过扩展上限时，优先拆分或抽取公共能力。
- 高相似度只代表“值得优先评估复用 / 重构”，不代表当前任务一定适合扩大重构。
- 搭配 `frontend-project-structure` 使用：结构规范决定“放哪里”，复用治理决定“是否应该新建”。
- Vue / React 均适用；具体复用单元名称跟随项目事实。

在以下场景必须使用：

- 新建或修改业务组件之前。
- 新建或修改通用组件之前。
- 新建 Vue composable、React hook、store、utils、API 封装、Mock 逻辑之前。
- 重构或审查重复实现、重复组件、重复业务逻辑时。
- 用户要求“复用治理 / 避免重复组件 / 避免重复造轮子 / 相似组件检查”。

## 技术栈上下文 + 三 skill 协同协议

当前端开发/修改/审查任务涉及页面、组件、API、Mock、样式或交互时，默认先读取项目技术栈上下文，再依次执行：

0. 技术栈上下文：`docs/frontend-tech-stack.md`、`package.json`、锁文件、构建/样式/路由/状态管理/UI 库等现有配置。
1. `frontend-project-structure`
2. `frontend-reuse-governance`
3. `frontend-code-style`

若用户只显式调用当前 skill，则只执行当前层，但最终报告必须提醒：**协同协议未完整执行**。

规则冲突时，优先级固定为：

1. 技术栈上下文 / 项目事实：技术选型、依赖边界、构建体系优先。
2. `frontend-project-structure`：目录结构、目录命名、文件命名次之。
3. `frontend-reuse-governance`：复用、扩展、重构与新建判断再次。
4. `frontend-code-style`：具体编码风格、错误处理、资源清理、交互完整性最后。

`FAIL` 是否阻塞取决于执行模式：生成模式默认阻塞；审查模式不阻塞报告输出；YOLO 模式记录原因、风险与后续建议后继续推进。若用户未指定模式，按任务意图判定并在输出中说明。

## 执行模式与 FAIL 处理策略

1. **生成模式**
   - 适用：新建页面、组件、API、Mock、store、样式、交互逻辑，或主动修改代码以交付功能。
   - FAIL 处理：在生成模式下阻塞交付；必须修复到 `PASS`，或记录用户确认例外。
   - 目的：防止新生成内容继续扩大技术债。
2. **审查模式**
   - 适用：用户要求检查、审查、列问题、list、评估规范符合度，但没有要求立即修改。
   - FAIL 处理：不阻塞报告输出；只列问题、影响、建议优先级与可选修复方案。
   - 目的：避免审查任务因为发现问题反而无法交付审查结论。
3. **YOLO 模式**
   - 适用：用户明确要求 `yolo`、快速推进、自行判断、不要等待确认、需要阻塞时根据上下文自行通过等场景。
   - FAIL 处理：遇到本应阻塞的 `FAIL` 时，记录判断原因、风险、影响范围和后续建议，然后继续推进；不得静默忽略。
   - 目的：减少等待用户决策造成的效率损耗，同时保留风险可追踪。
4. **模式默认判定**
   - 用户明确“生成 / 新建 / 实现 / 修改 / 修复”时，默认生成模式。
   - 用户明确“审查 / 检查 / list / 评估 / 不要修改”时，默认审查模式。
   - 用户明确“yolo / 自行通过 / 别等我 / 不用确认 / 快速推进”时，默认 YOLO 模式。
   - 无法判断时，先说明默认采用的模式；如果模式会显著改变是否阻塞，再询问用户。

## 强制执行流程

1. 先通过 `frontend-project-structure` 明确目标能力的结构归属：`business`、`business-shared`、`components`、`api`、`mock`、`stores` 等。
2. 识别目标资产类型：UI 组件、hook / composable、utils、store、API 封装、Mock 或其他业务逻辑。
3. 开发前按层级搜索默认范围：
   - 同 feature：`src/business/<domain>/<feature>/`。
   - domain shared：`src/business/<domain>/shared/`。
   - business-shared：`src/business-shared/`。
   - 全局通用组件：`src/components/`。
   - 逻辑资产：项目现有 `hooks`、`composables`、`stores`、`utils`、`api`、`mock` 等目录。
4. 先按资产类型判断复用重点，再用名称语义、视觉形态、功能行为、数据模型作为辅助线索。
5. 将候选项相似度判定为高 / 中 / 低，并说明依据。
6. 能小范围扩展现有实现时，优先扩展；超过扩展上限时，不继续堆 props / 参数，应拆分或抽取公共能力。
7. 高相似度时必须做重构风险评估，再决定重构、局部复用或新建并记录收敛建议。
8. 中 / 低相似度允许新建，但仍需说明候选项差异，并避免复制可抽取的小逻辑。
9. 完成后逐项自检；生成模式下有阻塞性 `FAIL` 不得声称完成，审查模式需列明问题，YOLO 模式需记录自行通过原因、风险与后续建议。

## 分层搜索范围

复用搜索必须按层级展开，避免只看当前 feature 或全局组件而漏掉最关键的业务共享层。

1. 单一 feature 内：`src/business/<domain>/<feature>/`。
2. 同 domain 跨 feature：`src/business/<domain>/shared/`。
3. 跨 domain 但仍有业务语义：`src/business-shared/`。
4. 彻底业务无关 UI：`src/components/`。
5. 逻辑资产：项目现有 `hooks`、`composables`、`stores`、`utils`、`api`、`mock` 等目录。

搜索时优先就近复用；只有出现真实跨层复用需求，才逐级上提。

## 按资产类型判断复用

不同资产的复用重点不同，不要把所有候选项都粗略套用同一套线索。

### UI 组件

重点看：

- 视觉结构是否相近，例如表格、筛选区、弹窗、状态标签、步骤条。
- 交互行为是否相近，例如查询、重置、编辑、删除、导入、导出。
- props / emits / callback、slot / children、状态边界是否能自然表达新需求。
- 样式定制是否属于合理配置，还是会把业务差异硬塞进通用组件。

### hook / composable

重点看：

- 输入、输出、返回状态和方法是否相近。
- Vue `watch` / 生命周期或 React `useEffect` / cleanup 等副作用是否一致。
- 错误处理、缓存策略、请求取消、资源清理是否符合当前场景。
- 是否混入过多 UI 组件细节，导致逻辑复用反而耦合。

### utils

重点看：

- 是否是纯函数，输入输出是否清晰稳定。
- 边界条件、空值、异常值处理是否一致。
- 是否带业务语义；带业务语义的 utils 不应误放到彻底通用层。

### store

重点看：

- 状态归属是全局应用级、单 domain，还是单 feature 内局部状态。
- action / selector / getter 是否表达同一业务行为。
- 是否影响跨页面缓存、刷新、权限、组织切换等全局行为。

### API 封装

重点看：

- endpoint、业务实体、请求参数和响应结构是否一致。
- 错误处理、请求拦截、分页、筛选、排序等封装是否可复用。
- 是否只是接口名字相似但业务语义不同，避免错误合并。

### Mock

重点看：

- 是否镜像对应 API 结构。
- 数据模型、状态枚举、场景覆盖是否一致。
- 随机数据生成逻辑、分页、筛选、异常场景是否可复用。

## 相似度等级

相似度是工程判断，不要求伪精确百分比。

### 高相似度

名称语义、核心行为、主要数据模型或视觉形态高度重叠。默认优先评估复用、扩展或重构，但必须结合重构风险决定是否扩大改动。

### 中相似度

部分行为或形态相似，但业务规则、数据模型或交互细节存在明显差异。允许新建，但需说明不能复用的关键差异；可抽取局部 hook / composable / utils。

### 低相似度

仅名称、局部 UI 或少量工具逻辑相似。允许新建，但仍需避免复制粘贴可抽取的小逻辑。

## 复用决策规则

### 1. 能小范围扩展再扩展

已有实现能通过少量 props、slot、children、render prop、配置项或函数参数覆盖新需求，且不会破坏现有调用方语义时，优先扩展现有实现。

禁止为了少量差异复制一份新组件或新逻辑。

### 2. 扩展上限

出现以下情况时，不建议继续扩展，应考虑拆分组件、抽取公共子组件、抽取 hook / composable / utils，或保持独立实现并记录原因：

- 为新场景新增多个互斥 props / 配置分支。
- 组件内部出现大量 `if/else` 区分业务场景。
- 新参数只服务单一调用方，却污染公共 API。
- 扩展后组件名称已无法准确表达职责。
- 为兼容旧行为引入难以测试的组合状态。
- 扩展会破坏旧调用方、增加隐藏副作用，或显著提高理解成本。

### 3. 就近复用优先

复用层级按就近原则逐级上提：

1. 单一 feature 内复用：保留在 `src/business/<domain>/<feature>/`。
2. 同 domain 跨 feature 复用：上提到 `src/business/<domain>/shared/`。
3. 跨 domain 但仍有业务语义：上提到 `src/business-shared/`。
4. 彻底业务无关 UI：上提到 `src/components/`。

具体文件归属必须服从 `frontend-project-structure`。

### 4. 高相似度处理

高相似度候选应优先评估以下路径：

- 直接复用现有实现。
- 小范围扩展现有实现。
- 抽取共享子组件、hook / composable、utils 或配置。
- 在风险可控时重构适配。

不得因为相似度高就自动扩大重构；必须先完成重构风险评估。

### 5. 重构风险评估

重构前至少评估：

- 调用方数量：调用方越多，重构影响越大。
- 回归风险：是否影响核心流程、权限、表单提交、数据变更等高风险路径。
- 测试覆盖：是否有单元测试、E2E 或可手动验证路径。
- 交付紧急程度：当前任务是否允许扩大变更范围。
- 依赖稳定性：候选实现是否稳定，是否本身质量较差。

决策规则：

- 高相似 + 低风险：优先重构或抽取复用。
- 高相似 + 高风险：允许暂时新建或局部复用，但必须记录原因和后续收敛建议。
- 中相似：通常优先局部复用或抽取小逻辑，不强制重构。
- 低相似：允许新建，避免复制可抽取的小逻辑。

### 6. 避免过早公共化

如果只是“未来可能复用”，不要直接放入全局公共层。

应先放在当前 feature 或 domain shared 中，等出现真实跨域复用后再上提。

### 7. 允许新建的常见理由

以下情况可以新建，但必须在交付说明中记录：

- 相似候选仅视觉接近，但业务行为和数据模型明显不同。
- 扩展现有实现会造成参数爆炸、万能组件或破坏旧调用方。
- 重构影响范围过大、测试覆盖不足、回归风险高，当前任务不适合扩大变更。
- 用户明确要求保持独立实现。
- 现有实现质量过低，复用会扩大技术债，且已说明后续收敛建议。

## 通用辅助线索

在完成资产类型判断后，可用以下线索辅助判断相似度：

- 名称语义：文件名、组件名、函数名、变量名、业务实体名是否表达相近能力。
- 视觉形态：UI 形态是否接近，例如表格、筛选区、弹窗、状态标签。
- 功能行为：查询、筛选、重置、新建、编辑、删除、详情、导入导出等行为是否重叠。
- 数据模型：props、emits、callback、API 入参、响应字段、状态枚举、表格列是否类似。

## 禁止事项

- 禁止开发前不搜索就新建业务组件、通用组件或业务逻辑。
- 禁止只搜索同 feature、全局组件和逻辑资产，漏掉 domain shared 与 business-shared。
- 禁止复制已有表格、筛选区、弹窗、状态标签等近似组件后只改少量字段。
- 禁止复制多份状态枚举、状态文案、状态颜色映射。
- 禁止复制近似 API 封装、Mock 数据生成逻辑或请求参数组装逻辑。
- 禁止把应在同 feature 或 domain shared 复用的能力直接塞到全局 `components`。
- 禁止为了“未来可能复用”过早抽象高复杂度公共组件。
- 禁止为复用继续堆叠互斥 props / 参数，把组件或逻辑变成万能怪物。
- 禁止高相似度时无风险评估就直接扩大重构。
