# 数据汇聚模块前端代码生成规范

## 1. 技术栈（不得偏离）

- UI框架：Vue 3 + TypeScript
- 构建工具：Vite
- 路由：vue-router，使用 unplugin-vue-router 实现基于文件的路由（File-based Routing）
  - 页面文件放在 src/pages/ 目录下，路由由插件自动生成，禁止手写 RouteRecordRaw 数组
  - 参考：https://router.vuejs.org/zh/file-based-routing/
- 状态管理：pinia
- 组件库：Element Plus + @element-plus/icons-vue
- 图表：echarts / echarts-gl / echarts-liquidfill（按需）
- 图标：@iconify-json/mdi + UnoCSS presetIcons，用法：<i class="i-mdi-xxx" />
- 原子化样式：UnoCSS
- 预处理器：Less
- 包管理：pnpm
- 类型检查：vue-tsc

---

## 2. 目录结构规范

```
src/
  pages/collect/          ← 路由页面入口（unplugin-vue-router 约定目录）
    datasource/
      index.vue           ← 数据源列表页（只做布局组装，不写业务细节）
      [id].vue            ← 数据源详情页
      config/[id].vue     ← 数据源配置页
    task/
      index.vue
      [id].vue
    dag/
      index.vue
      [id].vue
    extract/
      index.vue
    log/
      index.vue
    exec/
      [id].vue
  features/collect/       ← 业务组件与逻辑
    datasource/
    task/
    dag/
    extract/
    exec/
  components/             ← 通用无业务UI组件
  api/collect/
    index.ts              ← 接口函数 + 完整 TypeScript 类型定义
  mock/collect/
    index.ts
  styles/
  core/                   ← Cesium、地图、3D场景等底层能力（本模块不涉及）
```

**原则**：
- `src/pages/` 只做页面入口和 `<feature组件>` 的组装，不堆积业务逻辑
- `src/features/` 放业务组件和 composable，不跨模块相互引用
- `src/components/` 放通用 UI 组件，不在此发起特定业务 API 请求
- `src/api/` 统一封装接口，不在组件中直接使用 fetch/axios

## 3. API 层规范（最重要，必须严格遵守）

### 3.1 类型定义

所有接口入参、响应数据必须定义 TypeScript 类型，放在 `src/api/collect/index.ts` 顶部：

```typescript
// 通用响应包装
export interface ApiResult<T = unknown> {
  code: number
  message: string
  data: T
}

// 分页响应
export interface PageResult<T> {
  total: number
  list: T[]
  pageNum: number
  pageSize: number
}

// 各业务实体类型（见下方详细要求）
export interface Datasource { ... }
export interface CollectTask { ... }
export interface CollectTaskExec { ... }
export interface ExtractConfig { ... }
```

Mock 数据中的所有字段（包括 `taskName`、`datasourceName`、`errorDetail` 等接口
响应中实际返回的扩展字段）必须在对应类型中声明，不允许出现类型里没有但 Mock 数据
或模板里用到的字段。

### 3.2 函数命名

### 3.3 BASE 常量

每个业务域必须在对应区块顶部定义独立的 BASE 常量，禁止跨区块复用或遗漏：

```typescript
const DS_BASE    = '/api/v1/collect/datasource'
const TASK_BASE  = '/api/v1/collect/task'
const EXEC_BASE  = '/api/v1/collect/exec'
const EXTRACT_BASE = '/api/v1/collect/extract'
const DAG_BASE   = '/api/v1/collect/dag'
```

### 3.4 接口完整性自检

每次生成完 api/collect/index.ts 后，必须执行以下自检：
- 遍历所有组件中的 import { xxx } from '@/api/collect'，确认每个 xxx 在 api 文件中都有对应的具名导出
- 所有 DAG 相关函数使用 DAG_BASE，不允许使用未定义变量

## 4. 组件开发规范

### 4.1 类型使用

- 禁止使用 `ref<any[]>()`，必须用具体类型，例如 `ref<Datasource[]>([])`
- 禁止使用 `row: any`，表格行数据应声明类型，例如 `{ row }: { row: CollectTask }`
- 弹窗 form 必须声明具体类型，不允许 `ref<Record<string, any>>({})`

### 4.2 弹窗表单重置

弹窗关闭时，`handleClose` 必须重置**所有** form 字段（包括扩展字段），
不得只重置部分字段。推荐做法：

```typescript
const initialForm = (): FormType => ({
  name: '',
  dsType: '',
  // ... 所有字段
})
const form = reactive(initialForm())

function handleClose() {
  Object.assign(form, initialForm())
  emit('update:visible', false)
}
```

### 4.3 异步操作错误处理

所有异步操作必须有错误处理，不允许静默吞错：

```typescript
// 正确：ElMessageBox.confirm 取消时不应报 uncaught，用 try/catch 统一处理
async function handleDelete(row: CollectTask) {
  try {
    await ElMessageBox.confirm(`确认删除「${row.name}」？`, '提示', { type: 'warning' })
    await deleteTask(row.id)
    ElMessage.success('删除成功')
    loadData()
  } catch {
    // 用户取消，静默处理
  }
}

// 正确：watch 内的异步调用
watch(() => form.taskType, async (val) => {
  if (!val) return
  try {
    const { data } = await getDatasourcePage({ dsType: ..., pageSize: 100 })
    datasourceOptions.value = data.list
  } catch {
    ElMessage.error('加载数据源失败')
    datasourceOptions.value = []
  }
})
```

### 4.4 防抖与资源清理

凡使用 setTimeout/setInterval，必须在 onUnmounted 中清理：

```typescript
import { onUnmounted } from 'vue'


let searchTimer: ReturnType<typeof setTimeout>
onUnmounted(() => clearTimeout(searchTimer))

// DAG 轮询同理
let pollTimer: ReturnType<typeof setInterval>
onUnmounted(() => clearInterval(pollTimer))
```

### 4.5 功能完整性要求

所有 UI 元素必须有对应的实现，禁止出现"空壳交互"：
- 搜索框必须绑定 v-model 并触发 loadData
- 筛选下拉必须绑定 v-model 并触发 loadData  
- 新建/编辑按钮必须绑定对应弹窗的 visible 状态
- 导出按钮如暂未实现，必须给出占位提示而不是空函数：
  ```typescript
  function handleExport() {
    ElMessage.info('导出功能待实现')
  }
  ```
- "编辑"和"详情"不能共用同一个路由跳转函数

## 5. Mock 数据规范

- Mock 数据中出现的所有字段，必须在对应 TypeScript 接口类型中声明
- 常见易遗漏字段清单（必须在类型和 Mock 中保持一致）：
  - `CollectTaskExec` 需包含：`taskName`、`taskType`（列表展示用）
  - `CollectTask` 需包含：`datasourceName`（列表展示用，关联数据源名称）
  - `CollectTaskExec` 需包含：`errorDetail`（执行详情页错误堆栈展示）
- DAG Mock 数据必须包含：`id`、`name`、`nodeCount`、`taskSummary`、
  `scheduleMode`、`cronExpr`、`lastRunTime`、`status`、`nodes`、`edges`

## 6. 代码规范

- 缩进：2个空格
- 引号：单引号，无分号（Prettier 默认：singleQuote: true, semi: false）
- 行宽：printWidth: 100
- 组件文件：PascalCase，如 `DatasourceList.vue`
- 注释：优先中文，说明"为什么"而非"做什么"
- 布尔变量：is/has/can 前缀
- 事件处理函数：handle 前缀
- 状态/映射辅助函数（如 statusType、statusText）：提取为独立函数或 computed，
  多个组件用到的提取到 features/collect/shared/ 下的 composable，
  禁止在每个组件中各写一遍相同逻辑

---

## 7. 注释禁忌

- 禁止在注释或文件头中使用无意义缩写代号，如 `(pg-ds)`、`(newDsOv)` 等
- 正确示例：`// 数据源管理列表页 — 支持按类型/状态筛选、关键词搜索`
- 复杂业务规则、非直觉参数必须注释说明原因
- 样板代码（简单 v-model 绑定、基础条件判断）不需要注释
